# RFC: Onboarding Platform

**Author:** Carol Donnelly
**Status:** In progress
**Created:** 2026-03-28

## Problem

PostHog's onboarding process spans 6+ tools (Gmail, Google Sheets, GitHub Issues, Deel, DocuSign, Zluri, Make, Slack). While Zluri and Make handle provisioning and automation, much of the coordination between these tools is manual. Key pain points:

- **No single source of truth.** Hire status lives across a Google Sheet tracker, GitHub issue checklists, Deel contract status, and Slack conversations.
- **Passive task delivery.** The GitHub issue checklist tags people and hopes they notice/complete. No reminders, no escalation, no visibility into what's overdue.
- **Manual country/role logic.** Contract routing (US vs UK vs Germany vs international), provisioning playbooks (Engineering vs Sales), and compliance tasks (I-9, right to work) are tracked manually by the Ops team.
- **No contract monitoring.** For international hires on Deel, Ops manually checks multiple times over several days to see if the contract was signed. No notification exists.
- **Manual offer and contract creation.** Talent builds offers in a Google Sheet calculator, then manually communicates details to Ops via email. Candidate information is re-entered across multiple systems (Deel, DocuSign, the tracker), creating opportunities for data errors in compensation figures, contract details, and start dates. 

## Motivation

The current process works but doesn't scale. The ops-platform already handles pay reviews, org charts, keeper tests, and salary sync. Extending it to own the full onboarding lifecycle is a natural evolution.

The goal is: **a new hire opens their my employee page and thinks "this company was expecting me."** Behind the scenes, every task is tracked, every deadline is enforced, and both the Ops and Talent teams have full visibility without manual coordination.

## Proposal

Build the onboarding workflow into ops.posthog.dev as a series of connected features. Each feature is independently valuable and builds on the previous.

### Vision: Three Connected Portals

1. **Ops Command Center** (for the Ops and Talent teams) — pipeline view, task tracking, offer builder, contract monitoring. The brain that orchestrates everything.

2. **New Hire Portal** (for the person joining PostHog) — personalized page with their checklist, key contacts, timeline, and PostHog context. 

3. **Manager Portal** (for team leads) — lightweight view of incoming hires, their responsibilities, and completion tracking. Slack nudges when tasks are due.

### Technical Approach

- Built as new routes/components in the existing ops-platform (TanStack Start, Prisma, PostgreSQL)
- New Prisma models: `OnboardingRecord`, `OnboardingTask`
- Task templates defined in code (not DB) for version control and type safety
- Server functions use `createAdminFn` (write operations) and `createOrgChartFn` (read-only views)
- CSV import follows existing CommissionImportPanel pattern (client-side PapaParse, preview table, idempotent upsert)
- Slack notifications follow existing `notifyOverduePerformanceProgramChecklistItems` pattern
- Test coverage infrastructure added alongside the feature ([#301](https://github.com/PostHog/ops-platform/pull/301)) — Prisma mocks, auth mocks, render helpers, and ~70 test files covering units, routes, and components

## What Success Looks Like

- A new hire's onboarding can be initiated and tracked entirely from the platform
- The right people are told the right things at the right time automatically
- Country-specific and role-specific logic is encoded in the system, not dependent on institutional knowledge
- The Google Sheet tracker, GitHub issue checklist, and salary calculator spreadsheet are all retired
- A new hire opens their portal and thinks "this company was expecting me"

## Open Questions

- **Deel API rate limits:** 5 req/sec org-wide. Need to be careful with contract monitoring polling.
- **Make migration:** When to sunset Make scenarios vs run them in parallel with platform-native features?
